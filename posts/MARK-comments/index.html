Leveraging MARK comments to organize your iOS project

When attempting to maintain our codebase it’s a good idea to have a clear and consistent way of structuring our code so it remains readable and easy to navigate. This becomes increasingly important as our codebase grows larger in size. To achieve this there are several approaches and tools available to us but the use of MARK comments is the one I want to highlight with this post.

What is a MARK comment and how do we use them?

MARK comments are comments with “special abilities” that are interpreted differently by Xcode compared to regular comments. They serve as a kind of labels within our code to help us organize it into sections to keep it easy to navigate and readable.

To add only a section label:

// MARK: Content View

// MARK: -

Adds only a dividing line

// MARK: - My Section Name

Adds a section label and a dividing line

Conclusion

Structuring our code is always a good idea, especially when the files grow large. Keep in mind, though, that large files often indicate that a refactor may be in order to split up the responsibilities into smaller ones. We should also not go overboard and start putting MARK comments on every other line, that would just clutter our files and defeat its purpose. Ultimately - as with so many things - it’s up to you and your team to find the right balance of how to use them.

Personally I try to avoid merging code containing FIXME or TODO comments, as my experience is that they tend to be easily forgotten. If you find yourself in a situation where you really need to merge your unfinished code (we all know it happens) with these types of comments, I highly recommend adding a #warning("") statement next to them to have the compiler help remind you about their existence.
